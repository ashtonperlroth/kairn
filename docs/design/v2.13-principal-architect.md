# v2.13.0 — Principal-as-Architect (Creative Evolution)

> The proposer is gradient descent with a tiny learning rate. The Architect is simulated annealing.

## Problem

The current evolution proposer (`src/evolve/proposer.ts`) is constrained by design:

1. **Reactive only.** Every mutation must be "tied to a specific trace observation." It cannot propose something speculative.
2. **Conservative.** Max 3 mutations per iteration. Can't restructure systems — only patch text.
3. **Trace-grounded.** Proposes fixes for failures it *saw*. Cannot improve capabilities that no task tested.
4. **No cross-pollination.** Each evolve run is isolated. Discoveries in one project don't transfer to another.

This means the proposer finds the nearest local minimum in harness-quality space and plateaus. It will converge at ~75% on most projects, and the remaining 25% requires creative structural changes it can't make.

**Physics analogy:** The proposer does gradient descent in a high-dimensional loss landscape. It follows the local gradient but can't jump between basins of attraction. For that, you need exploration mechanisms — temperature (simulated annealing), crossover (genetic algorithms), or vision (a fundamentally different optimization objective).

## Architecture

### The Three-Mode Evolution Loop

```
Iteration 1-4:  REACTIVE PROPOSER (current — fix what's broken)
Iteration 5:    ARCHITECT (new — reimagine the structure)
Iteration 6-9:  REACTIVE PROPOSER (refine the architect's changes)
Iteration 10:   ARCHITECT (another structural pass)
...repeat...
```

The architect runs every Nth iteration (configurable). Between architect iterations, the reactive proposer fine-tunes. This is **epsilon-greedy at the iteration level**: mostly exploit (reactive), occasionally explore (architect).

### Architect Proposer

**System prompt (fundamentally different from reactive proposer):**

```
You are an expert agent environment ARCHITECT. Your job is NOT to fix bugs — 
the reactive proposer handles that. Your job is to REIMAGINE the harness structure 
for maximum agent effectiveness.

## What You Have Access To
1. Current harness files
2. Execution traces (for context, not root-cause diagnosis)
3. Evolution history (what's been tried, what worked)
4. Knowledge base (patterns that worked in other projects)

## Your Task
Step back from individual task failures. Ask:
- Is the overall harness ARCHITECTURE sound?
- Are there missing capabilities that no task exposed?
- Are there structural patterns from other projects that would help here?
- Should files be split, merged, reorganized, or replaced entirely?
- Are there hooks, rules, or agents that should exist but don't?

## Budget
You may propose up to 10 mutations (vs. reactive proposer's 3).
Structural changes require more mutations to be coherent.

## Rules
- You may create new files, delete files, and restructure significantly.
- You must preserve what's working — check scores before proposing removals.
- Each mutation needs a rationale, but it can be SPECULATIVE ("I believe X will improve Y because Z pattern works in similar projects").
- Bold changes are preferred over incremental tweaks. The reactive proposer handles tweaks.
```

**Key differences from reactive proposer:**

| | Reactive Proposer | Architect |
|--|---|---|
| **Trigger** | Every iteration | Every Nth iteration |
| **Mutations** | Max 3 | Max 10 |
| **Rationale** | Must cite specific trace failure | Can be speculative |
| **Objective** | Fix what's broken | Improve what's possible |
| **Knowledge** | Current run only | Cross-run knowledge base |
| **Risk tolerance** | Conservative (rollback on regression) | Moderate (staging before acceptance) |

### Staging & Acceptance

Architect proposals are risky — they change structure, not just content. To prevent catastrophic regressions:

1. Architect proposes mutations → applied to a **staging harness** (copy of current best)
2. Staging harness evaluated on **full task suite** (no pruning, no sampling)
3. If staging score ≥ current best score: accept → becomes new current harness
4. If staging score < current best score: reject → continue with current best
5. Rejected proposals logged in knowledge base as "tried, didn't help for this project"

This is like a **merge gate** — the architect can propose anything, but it only lands if it proves itself.

### Cross-Repo Research Protocol

**Command:** `kairn evolve research`

**Flow:**
1. **Select repos.** User provides a list of GitHub repo URLs, or Kairn searches for popular repos by language/framework.
2. **Clone & scan.** Clone each into a temp directory. Run `scanProject()` to profile.
3. **Generate or optimize.** If no `.claude/`, run `kairn describe -q -y`. If existing, run `kairn optimize -y`.
4. **Init evolve.** Run `kairn evolve init` with auto-generated tasks.
5. **Run evolve.** Run `kairn evolve run --iterations 10` (or configurable).
6. **Catalog mutations.** Extract all accepted mutations from all repos.
7. **Convergence analysis.** Identify mutations that appear in ≥M/N repos.
8. **Report.** Output a research report:

```markdown
## Universal Patterns (appeared in 8/10 repos)
- PostToolUse doc-update hook (mean score delta: +12%)
- Tech-stack-specific permissions (mean score delta: +8%)
- "Available Commands" section in CLAUDE.md replacing intent routing

## Language-Specific Patterns
### Python (3/3 Python repos)
- `Bash(pytest *)` in allow list
- ruff formatter hook

### Node.js (4/4 Node repos)  
- prettier PostToolUse hook
- `Bash(npm run *)` permissions

## Failed Experiments (tried in ≥3 repos, never helped)
- Adding memory.json persistence for simple projects
- Extra security rules beyond baseline 5
```

### Knowledge Base

**Location:** `~/.kairn/knowledge/`

**Structure:**
```
~/.kairn/knowledge/
├── patterns.jsonl          # All discovered patterns
├── convergence.json        # Cross-repo convergence analysis
└── projects/
    ├── inferix.json        # Per-project evolution history
    ├── my-api.json
    └── ...
```

**Pattern schema:**
```json
{
  "id": "pattern_abc123",
  "type": "universal|language|framework|project",
  "description": "PostToolUse prompt hook for doc updates",
  "mutation": { "file": "...", "action": "add_section", "new_text": "..." },
  "evidence": {
    "repos_tested": 10,
    "repos_helped": 8,
    "mean_score_delta": 12.3,
    "languages": ["Python", "TypeScript", "Go"]
  },
  "discovered_at": "2026-04-03T...",
  "last_validated": "2026-04-10T..."
}
```

The proposer and architect both read `patterns.jsonl` before proposing. Universal patterns with strong evidence are suggested proactively. Failed patterns are avoided.

### Exploration/Exploitation Schedule

**Default: `explore-exploit`**

```
Iterations 1-2:   ARCHITECT (exploration — bold structural changes)
Iterations 3-7:   REACTIVE PROPOSER (exploitation — targeted fixes)
Iteration 8:      ARCHITECT (mid-run exploration)
Iterations 9-N-1: REACTIVE PROPOSER (convergence)
Iteration N:      PRINCIPAL (synthesis of best mutations from entire run)
```

**Alternative: `constant`** — architect runs every Nth iteration with no schedule changes.

**Alternative: `adaptive`** — switch to architect mode when reactive proposer plateaus (score hasn't improved in 2+ iterations).
