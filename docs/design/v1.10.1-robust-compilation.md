# v1.10.1 Design Doc — Robust Compilation (Subagent Architecture)

> Fix the JSON truncation failure that causes `kairn describe` to crash on complex prompts. Decompose the single monolithic LLM call into a multi-pass pipeline.

**Status:** Planned  
**Depends on:** v1.10.0  
**Priority:** Critical — affects ~60% of edge case prompts (4/6 complex scenarios fail)

---

## The Problem

`kairn describe` currently makes a **single LLM call** that must return the entire `EnvironmentSpec` as one JSON blob. This blob includes:
- `claude_md` (up to 120 lines of markdown embedded as a JSON string)
- `commands` (10-15 command definitions, each multi-line markdown)
- `rules`, `skills`, `agents`, `docs` (all inline)
- `tools` (selection + reasoning)
- `settings` (permissions, hooks)

For complex prompts (biotech, k8s, ML training, music production), the LLM's response exceeds the `max_tokens: 8192` limit or produces malformed JSON due to the sheer volume of content. The JSON gets truncated mid-value, and `JSON.parse()` throws.

**Error pattern:**
```
Failed to parse LLM response as JSON: Expected ',' or '}' after 
property value in JSON at position ~23000-26000 (line ~135-158)
```

**Test results (10 edge cases):**
- ✅ legal-contract (moderate complexity) — PASS
- ❌ biotech-lab — FAIL (truncated at position 22997)
- ❌ music-production — FAIL (truncated at position 23340)
- ❌ k8s-platform — FAIL (truncated at position 26137)
- ❌ ml-training — FAIL (truncated at position 25194)

---

## Root Cause Analysis

1. **`max_tokens: 8192`** in `src/compiler/compile.ts` line 101/128 — the EnvironmentSpec JSON for complex projects easily exceeds this
2. **Monolithic output** — the system prompt (`src/compiler/prompt.ts`) asks for everything in a single JSON response: tools + claude_md + commands + rules + skills + agents + docs + settings
3. **Embedded markdown in JSON** — `claude_md` and command contents are multi-line markdown strings inside JSON, which is extremely token-expensive (every newline, quote, and backslash must be escaped)
4. **No retry or continuation** — if the response is truncated, compilation fails immediately with no recovery

---

## The Fix: Multi-Pass Compilation Pipeline

Split the single monolithic LLM call into 3 sequential passes. Each pass has a focused scope and produces a manageable amount of JSON.

### Pass 1: Skeleton (tools + structure)

**Input:** User intent + tool registry  
**Output:** Small JSON with tool selection, project name, description, and a brief outline

```json
{
  "name": "proteomics-lab-tracker",
  "description": "...",
  "tools": [
    { "tool_id": "sequential-thinking", "reason": "..." },
    { "tool_id": "sqlite", "reason": "..." }
  ],
  "outline": {
    "tech_stack": ["Python", "pandas", "pyteomics"],
    "workflow_type": "data-analysis",
    "key_commands": ["ingest", "analyze", "report"],
    "custom_rules": ["data-integrity", "lab-protocols"],
    "custom_agents": ["data-reviewer"],
    "custom_skills": ["ms-data-analysis"]
  }
}
```

**Max tokens:** 2048 (plenty for this small output)  
**Why it works:** No embedded markdown. Just structured metadata.

### Pass 2: Harness Content (CLAUDE.md + commands + rules + agents)

**Input:** Pass 1 output (skeleton) + user intent  
**Output:** The harness content as separate fields

```json
{
  "claude_md": "# Proteomics Lab Tracker\n\n## Purpose\n...",
  "commands": {
    "help": "...",
    "ingest": "...",
    "analyze": "..."
  },
  "rules": {
    "continuity": "...",
    "security": "...",
    "data-integrity": "..."
  },
  "agents": {
    "qa-orchestrator": "...",
    "data-reviewer": "..."
  },
  "skills": {
    "ms-data-analysis/SKILL": "..."
  },
  "docs": {
    "TODO": "...",
    "DECISIONS": "...",
    "LEARNINGS": "...",
    "SPRINT": "..."
  }
}
```

**Max tokens:** 8192 (full budget for content, no tool selection overhead)  
**Why it works:** The LLM only generates content, guided by the skeleton from Pass 1.

### Pass 3: Settings + MCP Config (deterministic, no LLM needed)

This pass is **code, not LLM**. Generate `settings.json` and `.mcp.json` deterministically from the tool selections in Pass 1 and the registry data.

```typescript
function buildSettings(skeleton: Pass1Result, registry: RegistryTool[]): object {
  // Deterministic: permissions from tool types, hooks from workflow type
  return { permissions: { allow: [...], deny: [...] }, hooks: { ... } };
}

function buildMcpConfig(skeleton: Pass1Result, registry: RegistryTool[]): object {
  // Deterministic: look up each tool_id in registry, copy install.mcp_config
  return Object.fromEntries(
    skeleton.tools.map(t => {
      const reg = registry.find(r => r.id === t.tool_id);
      return [t.tool_id, reg?.install.mcp_config ?? {}];
    })
  );
}
```

**Why it works:** MCP configs and settings are already deterministic (they come from the registry). No reason to ask the LLM to generate them.

## Implementation Details

### Changes to `src/compiler/compile.ts`

The current `compile()` function:
```typescript
export async function compile(intent, onProgress): Promise<EnvironmentSpec> {
  const userMessage = buildUserMessage(intent, registry);
  const responseText = await callLLM(config, userMessage);  // ONE call
  const parsed = parseSpecResponse(responseText);
  return spec;
}
```

Becomes:
```typescript
export async function compile(intent, onProgress): Promise<EnvironmentSpec> {
  // Pass 1: Skeleton
  onProgress?.("Analyzing workflow...");
  const skeletonMsg = buildSkeletonMessage(intent, registry);
  const skeletonText = await callLLM(config, skeletonMsg, { maxTokens: 2048 });
  const skeleton = parseSkeletonResponse(skeletonText);

  // Pass 2: Harness content
  onProgress?.("Generating environment...");
  const harnessMsg = buildHarnessMessage(intent, skeleton);
  const harnessText = await callLLM(config, harnessMsg, { maxTokens: 8192 });
  const harness = parseHarnessResponse(harnessText);

  // Pass 3: Settings + MCP (deterministic)
  onProgress?.("Configuring tools...");
  const settings = buildSettings(skeleton, registry);
  const mcpConfig = buildMcpConfig(skeleton, registry);

  // Assemble
  return {
    id: `env_${crypto.randomUUID()}`,
    intent,
    created_at: new Date().toISOString(),
    name: skeleton.name,
    description: skeleton.description,
    autonomy_level: 1,
    tools: skeleton.tools,
    harness: {
      claude_md: harness.claude_md,
      settings,
      mcp_config: mcpConfig,
      commands: harness.commands,
      rules: harness.rules,
      skills: harness.skills ?? {},
      agents: harness.agents ?? {},
      docs: harness.docs,
    },
  };
}
```

### Changes to `src/compiler/prompt.ts`

Split `SYSTEM_PROMPT` into two focused prompts:

1. **`SKELETON_PROMPT`** (~80 lines) — Tool selection principles, output schema for skeleton only. No command/rule/agent content templates.

2. **`HARNESS_PROMPT`** (~200 lines) — CLAUDE.md template, command templates, rule templates, agent templates. Receives the skeleton as context. No tool registry (already decided).

This also reduces the system prompt size for each individual call, giving the LLM more room in its context window.

### Changes to `callLLM()`

Add `maxTokens` parameter:
```typescript
async function callLLM(
  config: KairnConfig, 
  userMessage: string, 
  options?: { maxTokens?: number, systemPrompt?: string }
): Promise<string> {
  const maxTokens = options?.maxTokens ?? 8192;
  const system = options?.systemPrompt ?? SYSTEM_PROMPT;
  // ... rest unchanged
}
```

### Retry Logic for Pass 2

If Pass 2 JSON parsing fails (the most likely failure point), retry once with a shorter prompt that says "Be more concise. Maximum 80 lines for claude_md. Maximum 5 commands."

```typescript
try {
  harness = parseHarnessResponse(harnessText);
} catch (err) {
  onProgress?.("Retrying with concise mode...");
  const retryMsg = buildHarnessMessage(intent, skeleton, { concise: true });
  const retryText = await callLLM(config, retryMsg, { maxTokens: 8192 });
  harness = parseHarnessResponse(retryText);
}
```

## What Changes for the User

**Nothing.** The CLI interface is identical. `kairn describe "..." --yes --quick` works exactly the same way. The spinner will show slightly different progress messages:
- "Analyzing workflow..." (Pass 1)
- "Generating environment..." (Pass 2)  
- "Configuring tools..." (Pass 3)

Total compilation time may increase slightly (2 LLM calls instead of 1), but the cheap model routing from v1.10.0 already handles Pass 1 efficiently.
