# v2.12.0 — Generation Quality

> First real-world test exposed 6 critical generation flaws. This version fixes them.

## Background

Running `kairn describe` on [inferix](~/Projects/inferix) (a Python/Docker ML inference service with `api/`, `sdk/`, `dashboard/`, `tests/`, `docker-compose.yml`) produced a `.claude/` environment that Claude Code itself flagged as problematic. The feedback identified what works (rules, safety hooks, PostCompact, slash commands) and what doesn't (hallucinated structure, false-positive intent routing, wrong permissions, empty docs, .env security contradiction).

**Root cause:** `describe` doesn't run the scanner. It asks an LLM to *guess* the project structure from the user's natural language description. For established repos with real code, this produces hallucinated CLAUDE.md content.

## Changes

### 1. Gate `describe` to `optimize` for existing repos

**Problem:** `describe` hallucinates project structure for existing codebases because it never runs the scanner.

**Solution:** Early detection in `describe` before the clarification flow. If the working directory contains signals of an existing codebase, offer to redirect to `optimize`.

**Detection heuristic:**
```
Has any of: package.json, pyproject.toml, requirements.txt, Cargo.toml, 
            go.mod, Gemfile, Dockerfile, docker-compose.yml, src/, lib/, app/, api/
AND the directory has >5 non-hidden files
```

**Behavior:**
```
  This looks like an existing project with source code.
  For the best results, use: kairn optimize
  (kairn describe is designed for new projects or greenfield descriptions)
  
  ? Run kairn optimize instead? [Y/n]
```

If confirmed, call `optimizeCommand.parseAsync()`. If declined, proceed with `describe`.

**Files:** `src/commands/describe.ts`

### 2. Intent routing overhaul

**Problem:** Regex patterns on single words ("test", "fix", "run", "help", "check") fire on normal conversation. Intent-learner auto-promotes broader patterns. Tier 2 LLM prompt runs on every message.

**Solution:** Replace all three layers with:

**A. CLAUDE.md routing section (portable):**
```markdown
## Available Commands
When the user explicitly asks to run a workflow, use the appropriate command:
- `/project:build` — Build the Docker image
- `/project:test` — Run the full test suite
...
Only route when the user's clear intent is to execute a workflow.
Never route questions, discussions, or code reviews.
```

Generated from IR commands by `renderClaudeMd()`.

**B. Optional prompt hook (Claude Code, autonomy ≥ 2):**
Strict classification prompt in `settings.json` UserPromptSubmit. Only fires when autonomy level warrants it. Does NOT use regex.

**C. Remove entirely:**
- `hooks/intent-router.mjs`
- `hooks/intent-learner.mjs`  
- `hooks/intent-log.jsonl`
- `hooks/intent-promotions.jsonl`
- Tier 2 LLM prompt from UserPromptSubmit
- SessionStart learner hook
- `src/intent/` directory (patterns.ts, prompt-template.ts, router-template.ts, learner-template.ts, types.ts)

**Files:** `src/compiler/compile.ts`, `src/ir/renderer.ts`, `src/adapter/claude-code.ts`, `src/intent/*` (delete)

### 3. Tech-stack-aware permissions

**Problem:** `buildSettings()` hardcodes `Bash(npm run *)`, `Bash(npx *)` regardless of tech stack.

**Solution:** Derive from `skeleton.outline.tech_stack`:

| Tech Stack | Permissions |
|------------|------------|
| Python | `Bash(python *)`, `Bash(pip *)`, `Bash(pytest *)`, `Bash(uv *)` |
| Node/TS/JS | `Bash(npm run *)`, `Bash(npx *)` |
| Rust | `Bash(cargo *)` |
| Go | `Bash(go *)` |
| Ruby | `Bash(bundle *)`, `Bash(rake *)` |
| Docker | `Bash(docker *)`, `Bash(docker compose *)` |

Also: Python projects get a PostToolUse ruff/black formatter hook (analogous to the existing prettier hook for JS).

**Files:** `src/compiler/compile.ts` → `buildSettings()`

### 4. Living docs

**Problem:** Generated DECISIONS.md, LEARNINGS.md, SPRINT.md contain only template placeholders ("(Add decisions here as they are made)"). These waste context.

**Solution:**
- **Don't write empty docs.** In `writeEnvironment()`, filter out doc files matching placeholder patterns.
- **Add PostToolUse prompt hook** nudging Claude to update docs after meaningful changes:
  ```json
  {
    "matcher": "Write|Edit",
    "hooks": [{
      "type": "prompt",
      "prompt": "If this change involves an architectural decision, debugging insight, or task completion, consider updating .claude/docs/. Only update if genuinely useful."
    }]
  }
  ```
- **Update @doc-writer prompt** to prefer no output over template filler.

**Files:** `src/adapter/claude-code.ts`, `src/compiler/compile.ts`, `src/compiler/agents/doc-writer.ts`

### 5. Compilation UX

**Problem:** Progress renderer shows `◐ Pass 1: Analyzing... [0s]` — minimal feedback during long waits.

**Solution:**
- Animated spinner frames (braille: `⠋ ⠙ ⠹ ⠸ ⠼ ⠴ ⠦ ⠧ ⠇ ⠏`)
- Cumulative elapsed timer at top
- Richer phase descriptions with agent names
- Estimated time remaining (from `estimateTime()` minus elapsed)

**Files:** `src/ui.ts`, `src/compiler/batch.ts`

### 6. Honest .env handling

**Problem:** SessionStart hook injects .env values while `Read(./.env)` is in deny list. Security contradiction.

**Solution:**
- Remove .env injection hook entirely from generated settings.json.
- If project uses env vars (from skeleton or scanner): don't add `Read(./.env)` to deny. Document expected env vars in CLAUDE.md.
- If project doesn't use env vars: keep `Read(./.env)` in deny. No injection hook.

**Files:** `src/compiler/compile.ts` → `buildSettings()`, `src/ir/renderer.ts`

## Testing

1. `kairn describe` in existing repo → detects and offers redirect
2. `kairn describe` in empty dir → proceeds normally
3. `kairn optimize` on inferix → accurate CLAUDE.md with real structure
4. Generated settings.json: no intent hooks, correct permissions for tech stack
5. Generated CLAUDE.md: has "Available Commands" section
6. No .env injection hook; deny rule conditional
7. Empty docs not written
8. Spinner animation works during compilation
9. All existing tests pass (with intent tests removed/updated)

## Migration

Existing environments generated before v2.12.0 are unaffected. Users can run `kairn optimize` to regenerate with the fixes.
