# v1.14.0 Design Doc — Completion Verification (Double-Confirmation Pattern)

> Agents call "done" too early. Add a structured verification gate before any task is marked complete, catching premature exits that waste user time.

**Status:** Planned  
**Depends on:** v1.13.0  
**Inspired by:** [Meta-Harness](https://yoonholee.com/meta-harness/) (Stanford IRIS Lab, March 2026) — their TerminalBench-2 agent uses a double-confirmation pattern with a checklist that forces self-review before the agent can mark a task as complete.

---

## The Problem

Agents frequently declare tasks "done" when they're not. Common failure modes:

1. **Incomplete implementation** — Agent builds the happy path but skips edge cases, error handling, or tests
2. **Broken state** — Agent made changes that pass locally but break CI, imports, or other files
3. **Misunderstood requirements** — Agent solved a different problem than what was asked
4. **Side effects** — Agent left debug logs, temp files, commented-out code, or modified unrelated files

In Meta-Harness's TerminalBench-2 agent, the first `task_complete` call triggers a checklist instead of actually completing. The agent must review its work against the original task, verify from multiple perspectives (test engineer, QA engineer, user), and call `task_complete` *again* to actually finish. This pattern caught enough premature exits to measurably improve benchmark scores.

Kairn already generates `/project:prove` (from v1.7) for manual verification. But that's opt-in — the agent doesn't use it unless told to. v1.14 makes verification structural: it's woven into every completion-adjacent command so agents can't skip it.

---

## The Fix: Completion Verification Gate

### Core Concept

When an agent signals "done" (via `/project:develop`, `/project:loop`, `/project:auto`, or any orchestrated workflow), the harness forces a self-review step before the task is actually marked complete.

This is NOT about adding more commands. It's about modifying the *existing* completion logic in generated commands to include a verification gate.

### The Verification Checklist

Injected into every orchestrating command's "completion" phase:

```markdown
## Completion Verification

Before marking this task complete, you MUST verify against this checklist.
Do not skip any item — premature completion wastes more time than verification.

### Requirements Check
- [ ] Re-read the ORIGINAL task description (not your interpretation)
- [ ] List each explicit requirement → confirm each is met with evidence
- [ ] List each implicit requirement (error handling, types, tests) → confirm or justify omission

### State Check
- [ ] Run the project's test suite: `{{TEST_COMMAND}}`
- [ ] Run the project's lint/typecheck: `{{LINT_COMMAND}}`
- [ ] `git diff --stat` — review every changed file. Are there unexpected changes?
- [ ] No debug logs, console.logs, commented-out code, or temp files left behind
- [ ] No unrelated files modified

### Perspective Check
Answer each in 1 sentence:
- **As the test engineer:** What's the most likely way this breaks in production?
- **As the code reviewer:** What would I flag in PR review?
- **As the user who requested this:** Does this actually solve my problem?

### Verdict
If ALL checks pass: proceed to mark complete.
If ANY check fails: fix the issue, then re-run this checklist.
```

## How It Integrates with Existing Commands

### `/project:develop` (v1.11)

Currently, Phase 5 (Verify) spawns `@fixer` if tests fail, then Phase 6 (Review) spawns `@grill`. But there's no final self-check after the review is addressed.

**Change:** Add Phase 7 — "Completion Gate":

```markdown
## Phase 7: COMPLETION GATE

Before declaring this feature complete:

1. Run the Completion Verification checklist above
2. Confirm all items pass
3. If any fail, loop back to the appropriate phase
4. Only after ALL items pass, proceed to documentation and delivery

Do NOT skip this phase. Premature completion is the #1 cause of rework.
```

### `/project:loop` (v1.9, Level 2)

Currently loops: plan → implement → test → iterate. The loop exits when tests pass.

**Change:** Add a verification step after tests pass:

```markdown
## Loop Exit Condition

Tests passing is NECESSARY but NOT SUFFICIENT for completion.
After tests pass, run the Completion Verification checklist.
Only exit the loop when both tests pass AND the checklist clears.
```

### `/project:auto` (v1.9, Level 3)

Currently: PM plans → loop executes in worktrees → PR delivery.

**Change:** Before PR creation, the worktree agent must run the Completion Verification checklist. The PR description should include the checklist results.

### `/project:autopilot` (v1.9, Level 4)

Currently runs with stop conditions. Add the Completion Verification as a mandatory stop condition check:

```markdown
## Stop Condition Evaluation

Before evaluating whether to stop:
1. Run Completion Verification checklist
2. Include checklist results in the stop condition evaluation
3. If verification fails, the stop condition is NOT met regardless of other criteria
```

## Implementation

### Changes to `src/compiler/prompts.ts`

Add the verification checklist as a reusable template block:

```typescript
const COMPLETION_VERIFICATION_TEMPLATE = `
## Completion Verification

Before marking this task complete, verify against this checklist.
Premature completion wastes more time than verification.

### Requirements Check
- [ ] Re-read the ORIGINAL task description (not your interpretation)
- [ ] Each explicit requirement is met with evidence (test output, diff, or screenshot)
- [ ] Each implicit requirement (error handling, types, tests) is addressed or justified

### State Check
- [ ] Test suite passes: \`{{TEST_COMMAND}}\`
- [ ] Lint/typecheck passes: \`{{LINT_COMMAND}}\`
- [ ] \`git diff --stat\` — every changed file is intentional
- [ ] No debug artifacts (console.log, TODO, commented-out code, temp files)

### Perspective Check (1 sentence each)
- **Test engineer:** Most likely production failure mode?
- **Code reviewer:** What would I flag in review?
- **Requesting user:** Does this solve my actual problem?

### Verdict
ALL pass → mark complete.  
ANY fail → fix first, then re-verify.
`;
```

The `{{TEST_COMMAND}}` and `{{LINT_COMMAND}}` placeholders are filled based on project type:

```typescript
function inferVerifyCommands(projectType: string, skeleton: Skeleton): VerifyCommands {
  const commands: VerifyCommands = {
    test: 'echo "no test command configured"',
    lint: 'echo "no lint command configured"',
  };

  if (skeleton.scripts?.test) {
    commands.test = `npm test`;  // or detected test script
  } else if (projectType.includes('python')) {
    commands.test = `pytest`;
  } else if (projectType.includes('rust')) {
    commands.test = `cargo test`;
  } else if (projectType.includes('go')) {
    commands.test = `go test ./...`;
  }

  if (skeleton.scripts?.lint) {
    commands.lint = `npm run lint`;
  } else if (projectType.includes('python')) {
    commands.lint = `ruff check .`;
  } else if (projectType.includes('rust')) {
    commands.lint = `cargo clippy`;
  } else if (projectType.includes('go')) {
    commands.lint = `go vet ./...`;
  } else if (projectType.includes('typescript')) {
    commands.lint = `npx tsc --noEmit`;
  }

  return commands;
}
```

### Changes to `src/compiler/commands.ts`

Modify the generation of orchestrating commands to inject the Completion Verification block.

For `/project:develop`:
```typescript
function generateDevelopCommand(skeleton: Skeleton, verifyCommands: VerifyCommands): Command {
  const verification = COMPLETION_VERIFICATION_TEMPLATE
    .replace('{{TEST_COMMAND}}', verifyCommands.test)
    .replace('{{LINT_COMMAND}}', verifyCommands.lint);

  // ... existing phases 1-6 ...
  // Append Phase 7
  const phase7 = `
## Phase 7: COMPLETION GATE

${verification}

Only after ALL items pass, proceed to document decisions in DECISIONS.md and deliver.
`;

  return { name: '/project:develop', content: existingContent + phase7 };
}
```

For `/project:loop`:
```typescript
// Modify the loop exit condition to include verification
const loopExitCondition = `
## Loop Exit Condition

Tests passing is NECESSARY but NOT SUFFICIENT.
After tests pass, run the Completion Verification:
${verification}
Only exit when both tests and verification pass.
`;
```

### Changes to `src/compiler/compile.ts`

In Pass 2, compute verify commands from the skeleton and pass them to command generation:

```typescript
const verifyCommands = inferVerifyCommands(skeleton.projectType, skeleton);
const commands = generateCommands(skeleton, config.autonomyLevel, verifyCommands);
```

### CLAUDE.md Section

Also inject a top-level principle into the CLAUDE.md:

```markdown
## Completion Standards

Never mark a task "done" without running the Completion Verification checklist.
Tests passing is necessary but not sufficient — also verify requirements coverage,
state cleanliness, and review your changes from the perspective of a test engineer,
code reviewer, and the user who requested the work.
```

This goes after the existing "Verification" section from v1.7 but adds the behavioral mandate.

## What the User Gets (Example)

### During `/project:develop`

After the implementation is tested and reviewed:

```
Phase 7: Completion Gate

Running Completion Verification...

Requirements Check:
  ✔ Original task: "Add JWT auth to /api/users endpoints"
  ✔ JWT middleware created (src/middleware/auth.ts)
  ✔ All 4 /api/users routes protected
  ✔ Tests added for authenticated + unauthenticated access
  ✔ Error responses include proper 401 status

State Check:
  ✔ npm test — 47 passed, 0 failed
  ✔ npx tsc --noEmit — no errors
  ✔ git diff --stat: 4 files changed (auth.ts, users.ts, auth.test.ts, package.json)
  ✔ No debug artifacts

Perspective Check:
  - Test engineer: Token expiry edge case could fail if system clock drifts
  - Code reviewer: JWT_SECRET should come from env, not hardcoded — WAIT, checking...
    → Fixed: JWT_SECRET reads from process.env.JWT_SECRET ✔
  - User: Yes, all user endpoints now require valid JWT

Verdict: ALL checks pass. Marking complete.
```

### Premature Exit Caught

```
Phase 7: Completion Gate

Requirements Check:
  ✔ Original task: "Add pagination to /api/posts"
  ✔ ?limit and ?offset query params added
  ✖ FAIL: Task also mentions "return total count in response headers"
    → X-Total-Count header not implemented

Verdict: FAIL — looping back to Phase 4 (Implement)
```
